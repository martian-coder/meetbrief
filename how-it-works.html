MeetBrief / How it works

From spoken word to
AI reply in under 2 seconds

A fully local pipeline: system audio captured on-device, transcribed by Whisper, turn-segmented by VAD, then sent to your chosen LLM. The response streams token-by-token to the overlay before you've finished listening.

Step by step

The complete workflow

1

Load your meeting context

Before the meeting starts, paste your agenda, your role, the other participants, your key objectives, and any important background into the context panel. This text becomes the cached system prompt for every AI call during the session.

Thanks to Anthropic's prompt caching, this context is uploaded once and stored server-side for up to 5 minutes. Subsequent calls reuse the cached prefix, cutting input token cost by ~90% and reducing time-to-first-token significantly.

With Ollama, the system prompt is simply prepended to every request — no caching overhead, but no cost either.

2

Capture system audio

When you click Start Session, MeetBrief begins capturing system audio — what's playing through your speakers or headphones. On macOS it uses a native subprocess that dumps PCM audio as a raw stream. On Windows, WASAPI loopback capture is used. Linux uses the microphone by default (with system audio possible via PulseAudio loopback).

Audio is captured as stereo PCM, mixed to mono, and chunked into ~100ms frames ready for the VAD pipeline.

3

Detect speech turns with Silero VAD

Each audio chunk is passed through Silero VAD (Voice Activity Detector), a lightweight neural model that runs in-process. It outputs a confidence score indicating whether the chunk contains speech.

MeetBrief accumulates spoken chunks into a buffer. When VAD detects ~1 second of sustained silence after a speech segment, it marks the end of a turn and triggers transcription. This prevents mid-sentence interruptions and avoids generating replies while someone is still talking.

4

Transcribe with local Whisper

The accumulated speech buffer is sent to Whisper, running entirely on your machine via @huggingface/transformers. The model converts the audio segment to text. With the translate task enabled (default), speech in any language is automatically output as English text — so the LLM always receives English regardless of the meeting's language.

MeetBrief maintains a rolling 30-second transcript window. Older turns are summarised and prepended as context rather than dropped entirely — ensuring the AI has the full narrative arc without exceeding the LLM's context window.

5

Stream the AI reply to the overlay

The transcript window is appended as the user message, combined with your pre-loaded system prompt, and sent to the LLM. The response is streamed token by token directly to the floating overlay window.

The overlay displays the reply as it streams — typically the first words appear within 800ms of the API call. You can read the beginning of the response while the rest is still generating.

The overlay is a frameless, always-on-top Electron BrowserWindow with transparent: true and setIgnoreMouseEvents(true) — it floats above any app (Zoom, Teams, Meet) without interfering with clicks.

Cost efficiency

Prompt caching keeps API costs low

Why this matters: In a 60-minute meeting, MeetBrief may make 30–50 API calls. Without caching, the large system prompt (your meeting context) is re-sent and re-billed on every call. With Anthropic's prompt caching, the static context is billed at 10% of the normal input cost after the first cache write — saving ~90% on input tokens for every subsequent call in the session.

Static context → system prompt (cached)

Your agenda, role, objectives, and background are stored in a cache_control: ephemeral content block. Once uploaded, Anthropic keeps it warm for up to 5 minutes and reuses it on subsequent requests.

Live transcript → user message (not cached)

The rolling transcript window changes every turn, so it's sent fresh each time. Only the dynamic part pays full input token cost — which is typically much shorter than the static context.

Typical cost per hour (Claude)

With a 2,000-token context and prompt caching active, a one-hour meeting with 40 AI calls typically costs under $0.10. Without caching it would be ~$0.80.

Ollama — zero per-call cost

Switching to LLM_PROVIDER=ollama uses your local hardware. No API key, no per-call billing. Latency depends on your GPU/CPU. A mid-range M2 MacBook handles small models comfortably.

Providers

Choose your stack

C

Anthropic Claude

Highest quality replies. Fastest time-to-first-token with prompt caching. Requires an Anthropic API key. Recommended for most users. Set LLM_PROVIDER=anthropic.

🦙

Ollama (local)

Runs on localhost:11434. Zero API cost, fully offline. Best for privacy-sensitive environments. Quality and latency depend on the model and your hardware. Set LLM_PROVIDER=ollama.

W

Whisper STT

Always runs locally — no external STT service. Choose between whisper-tiny (fast, lower accuracy) or whisper-small (recommended balance) via the WHISPER_MODEL env var.
